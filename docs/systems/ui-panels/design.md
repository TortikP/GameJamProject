# UI Panels — Design Document (seed)

**Статус:** Seed. Накопленные правила и решения, которые войдут в Spec 0 этой системы. Не сам спек — design memo, описывающий рамку.
**Обсуждали:** Андрей (требования, идея системы как отдельной от редактора, два контрола, can't-lose-UI правило), Никита (collapse-в-шапке, дискуссия про close-кнопку, отвергнута), Claude (раскладка, развилки).
**Связанное:**
- Главный потребитель: [`docs/systems/level-editor/design.md`](../level-editor/design.md). Spec 1 редактора зависит от готовности Spec 0 этой системы.
- Потенциальные потребители (не в scope сейчас): редакторы диалогов/скиллов, in-game UI (skill HUD, character info, поп-апы) — миграция отдельным решением после Spec 0.

---

## 1. Зачем отдельная система

Панели в проекте сейчас делает каждый сам: `PanelContainer` + `DraggablePanel.new()` миксин (drag есть), collapse — местами, по-разному, ad-hoc. Resize нет вообще. Семь панелей в Map Editor — семь немного разных реализаций. При появлении следующего редактора (диалогов, скиллов) повторим этот же boilerplate ещё раз.

Решение: вытащить «универсальное окно интерфейса» в отдельную систему. Один тип, один контракт, переиспользуется во всех редакторах и потенциально во всех in-game панелях. Любая фича панелей — состояние замочка, поведение при resize, persistence раскладки — добавляется один раз и работает везде.

Это **не «framework для будущего»**, это удаление существующего дублирования + закрытие конкретных дыр (resize, persistence, защита от потери интерфейса).

---

## 2. Цели и не-цели

**Цели:**
- Один базовый тип панели с встроенными drag, resize, collapse, lock.
- Два контрола в шапке у каждой панели: замочек и `−`/`+` (collapse-toggle).
- Persistence раскладки: позиция, размер, свёрнутость, Z-порядок. Узкое snapshot (см. §3).
- Защита от потери интерфейса: невозможно вытащить панель так, чтобы её нельзя было вернуть. См. §6.
- Per-screen/scene scope persistence: панель в `map_editor.tscn` помнит своё место отдельно от той же панели в `dialogue_editor.tscn`.

**Не-цели:**
- Close-кнопка. Закрытия панели нет совсем — только collapse в плашку. Решение по итогам обсуждения; альтернатива (close + восстановление через bottom dock) отвергнута Андреем.
- Tab/dock system как в Godot editor.
- Layout presets (save/load named layouts).
- Reset-кнопка («верни как было по дефолту»). При выполненных правилах §6 потерять интерфейс нельзя, reset не нужен.
- Persistence «session state» (какая палитра выбрана, какой таб открыт, скролл-позиция). Только layout. Если когда-нибудь понадобится — отдельная фича.
- Панели как полноценные ОС-окна (vmin/vmax/maximize). Это не window manager.
- Миграция существующих in-game панелей. Side-польза, отдельное решение.

## 3. Persistence: что входит в snapshot

**Узкое snapshot.** Только то, что юзер сделал руками с самой панелью:

- Позиция (x, y) в координатах viewport
- Размер (width, height)
- Свёрнутость (collapsed bool)
- Состояние замочка (locked bool)
- Z-порядок относительно других панелей в той же scene

**Что не сохраняется:**
- Выбор внутри панели (активная палитра, выбранный таб, активная волна, выбранный гекс)
- Скролл-позиция
- Любое содержательное состояние

Логика: «layout» — свойство панели, «session state» — свойство контента. Их жизненные циклы разные. При расширении игры in-game UI «гвоздями прибьётся» — там даже layout юзер не будет двигать; session state там вообще нерелевантен.

---

## 4. Persistence: как и где хранится

**Файл:** `user://layouts.cfg` (Godot ConfigFile). Один файл на всю игру.

**Структура:**
```ini
[map_editor.tscn::LayersPanel]
position = Vector2(20, 80)
size = Vector2(220, 400)
collapsed = false
locked = false
z_index = 5

[map_editor.tscn::WavePanel]
...

[dialogue_editor.tscn::ToolPanel]
...
```

**Ключ:** `<scene_path>::<panel_id>`. `panel_id` — стабильный идентификатор панели (имя ноды или явный `panel_id` экспорт-параметр). Разные сцены = разные раскладки. Одна и та же панель в одной сцене = один layout.

**Когда сохраняется:** debounced autosave 0.5s после последнего изменения (drag-end, resize-end, collapse toggle, lock toggle, z-change). Совпадает с паттерном autosave карты.

**Когда загружается:** при `_ready` панели. Если ключа в файле нет — используется дефолтное расположение, заданное в сцене.

**Валидация при загрузке:** см. §6 (clamps).

**Версионирование:** в начале файла секция `[meta] version = N`. При изменении формата — миграция или сброс с инфо-логом. Старт со `version = 1`.

## 5. Контролы шапки

Два контрола, фиксированный порядок: `[🔒] [−]` (замочек слева, collapse-toggle справа). Заголовок панели — между или слева, не блокирует контролы.

**Замочек** — toggle:
- `unlocked` (default) → панель полностью интерактивна.
- `locked` → drag и resize заблокированы. Collapse работает. Замочек разлочивается тем же кликом.

Поведение «locked» одинаковое в редакторах и в in-game. Разработчик может задать `locked = true` дефолтом для in-game панелей через экспорт-параметр сцены — это и есть механизм «прибить интерфейс гвоздями» (см. §2 не-цели и §1 контекст). Юзер технически может разлочить, но кликнуть надо явно.

**`−` / `+`** — collapse toggle:
- Развёрнутая панель: показывает `[−]`. Клик → панель сворачивается в плашку с заголовком и контролами; контент скрывается; высота = высота шапки.
- Свёрнутая плашка: показывает `[+]`. Клик → разворачивает обратно к запомненному размеру до collapse.
- Свёрнутую панель можно драгать и (если разлочена) — но не resize'ить (нечего ресайзить).
- Размер до collapse запоминается отдельно от текущего, чтобы expand вернул именно прошлый размер.

Никаких других контролов в шапке нет. Никакого минимизировать-в-таскбар, dock, pin, sticky.

## 6. Защита от потери интерфейса (clamps)

Пять правил, обязательны для имплементации Spec 0. Все приняты Андреем.

**C1. Min size.** Панель нельзя ужать меньше `(min_width, min_height)` где min — размер, в который влазит шапка целиком (заголовок + замочек + collapse-кнопка). Ориентир — 120×32px, точное значение по реальному измерению шапки.

**C2. Bounds-clamp на drag.** При drag шапка панели обязана целиком оставаться внутри viewport. Тело панели может частично вылезать за края (это удобно — прижать панель к нижнему краю), но шапка с двумя контролами — нет. Это даёт возможность всегда «схватить» панель за шапку.

**C3. Clamp при resize окна / смене монитора.** При изменении размера viewport — на следующем кадре все панели проверяются: если шапка оказалась снаружи, позиция корректируется, чтобы шапка была внутри. Не сохраняем во время самого ресайза окна (юзер дёргает уголок — много событий), сохраняем после settling.

**C4. Clamp при загрузке layout.** При `_ready` — позиция из `user://layouts.cfg` валидируется против текущего viewport. Если вне границ — clamp. Перезаписываем файл новой валидной позицией.

**C5. Нет `mouse_filter = ignore` на панели.** Панель всегда interactive. Невозможно случайно сделать её «прозрачной для мыши» через конфигурацию.

При выполнении C1–C5 панель невозможно потерять. Reset-кнопка не нужна.

## 7. Resize: что было и что делаем

**Контекст:** в spec 055 (level editor predecessor, дропнут) resize попробовали и выпилили. Причина была не «не нужен» — реализация была сломана: невидимые HSIZE/VSIZE handles на краях создавали cursor surprises при наведении на края → accidental resize'ы во время быстрой работы. Андрей подтвердил: «Он точно нужен», берём с D-вариантом (full flexibility).

**Что делаем в Spec 0:**
- 8 handles: 4 угла + 4 стороны.
- Handles **видимые** в hover state (минимум — точечные индикаторы в углах + полоски на серединах сторон). Перманентно видимые — open question §9.
- Зона захвата handle ≥10px (чтобы не было «попал/не попал»).
- Cursor change (HSIZE/VSIZE/FDIAGSIZE/BDIAGSIZE) **только** когда мышь над зоной захвата handle. Не на всём краю.
- Min size enforce'ится на resize (см. C1).
- Bounds-clamp на resize **не** делаем — resize не может отправить шапку за пределы (она остаётся на месте при ресайзе bottom/right; при ресайзе top/left — шапка сдвигается вместе и всё ещё в viewport если изначально была).

В свёрнутом состоянии resize отключён (нечего ресайзить).

При locked = true resize отключён, handles не отрисовываются.

## 8. Z-order

При клике в панель она поднимается наверх (стандартное window поведение). Z-order сохраняется в layout.

Свёрнутая плашка не выскакивает на верх автоматически — остаётся там, где была, чтобы не «вылетать» в фокус случайно при разворачивании. Клик по `[+]` разворачивает + поднимает на верх.

При load: восстанавливаем относительный порядок панелей по сохранённым z_index. Если каких-то панелей нет в файле — они уходят на низ.

## 9. Открытые вопросы

Не блокеры на старт Spec 0, решаются по ходу:

- **UP-1: Перманентно видимые resize handles vs только в hover.** Hover-only чище визуально, но юзер может не знать что resize есть. Перманентно видимые — учат, но визуальный шум. Решать на прототипе Spec 0.
- **UP-2: Дефолтная позиция/размер новой панели, у которой ещё нет записи в layouts.cfg.** Берётся из сцены (`Control.position` / `Control.size`). Кто отвечает за начальные значения — разработчик панели в `.tscn`. Если несколько панелей по дефолту падают в одно место — это проблема дефолтов, не системы. Не требует решения, документируется как контракт разработчика панели.
- **UP-3: Должна ли панель помнить свой `unlock` через сессии?** Сейчас locked сохраняется. Альтернатива — дефолт `locked = false` (или из сцены) каждую сессию, юзер может только временно разлочить. Дефолт сейчас — сохраняем, как остальное состояние.
- **UP-4: Конфликт ключа в layouts.cfg при переименовании панели.** Если Spec 2 переименовывает `FloorPalettePanel → LayersPanel`, у юзеров в `layouts.cfg` старый ключ останется как мусор. Решение для Spec 0 — игнорировать неизвестные ключи. Чистка при крупных миграциях — отдельная задача.
- **UP-5: Multi-monitor и перенос окна между мониторами.** C3 покрывает основной случай (viewport уменьшился, clamp). Но specifically «окно переехало на монитор 2 с другим разрешением» — отдельный edge case. Считаем, что C3 этого достаточно; реальные баги — после плейтестов.

---

## 10. Не-цели (явный список против scope creep)

- Close-кнопка панели. Не существует.
- Reset-кнопка / «verni как было». Не существует.
- Layout presets (named layouts). Не сейчас.
- Persistence session state (выбор внутри панели). Не сейчас.
- Tabs/dock system. Не сейчас, не overkill.
- Pin / sticky / always-on-top как отдельные states (locked это не покрывает; добавим если попросят).
- Вкладочный bottom-dock закрытых панелей (Никитин концепт). Отвергнут — закрытия нет.
- Keyboard navigation между панелями (отложено). Tab/Alt+Tab между панелями — нет.
- Animation/easing на drag/resize. Снэп без анимации, минимум визуального шума.
- Темизация панелей. Берётся из общего theme проекта, не отдельная тема для ui-panels.
