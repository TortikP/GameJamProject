# Game Design Docs

Эта папка — **долговременная память дизайна игры**, отдельная от технической документации (`docs/`) и от спеков фич (`specs/`).

## Зачем существует

В чатах, питчах и обсуждениях рождается куча решений, которые потом теряются. Через две недели никто не помнит почему "аспекты", а не "стихии", и спор начинается заново. Эти доки — фиксация того, что уже решено, и что ещё открыто, чтобы не переоткрывать одно и то же.

## Что где лежит

| Файл | Что внутри |
|---|---|
| [`VISION.md`](VISION.md) | Однопараграфный elevator pitch: что игра есть и чего она не есть. |
| [`PILLARS.md`](PILLARS.md) | 3–5 принципов, которые игра не нарушает. Фильтр для фич. |
| [`aspects.md`](aspects.md) | Концепция аспектов как идентичности игрока — зафиксированная абстракция. |
| [`DECISIONS.md`](DECISIONS.md) | Журнал ратифицированных дизайн-решений. Дата + контекст + кто обсуждал. |
| [`OPEN-QUESTIONS.md`](OPEN-QUESTIONS.md) | Незакрытые дизайн-вопросы. То, что висит и ждёт решения. |
| [`REFERENCES.md`](REFERENCES.md) | Оммажи, тональные референсы, "как у X". |
| [`GLOSSARY.md`](GLOSSARY.md) | Единая терминология. Аспекты, билды, секторы, волны. |
| [`MAINTAINER.md`](MAINTAINER.md) | Роль агента, который следит за этими доками. |

## Конвенции

**"Обсуждали:" вместо "Овнер:"** — в этих доках и в спеках по умолчанию **не пишем овнеров фич**. Пишем тех, кто придумывал и обтёсывал идею, с маркером `(идея)` для автора. Овнерство уточняется в момент имплементации, не в момент дизайна. Пример:

```
Обсуждали: Сергей (идея), Андрей, Никита
```

**Решения в формате DECISIONS.md** — каждое решение получает одну запись с датой, формулировкой, контекстом и кредитом. Не пересматривается без явной отмены (тоже записывается).

**Открытые вопросы — в OPEN-QUESTIONS.md** — если решение ещё не принято, вопрос висит здесь с пометкой "ждёт чего/кого". Когда закрывается — переезжает в DECISIONS.md.

**Доки живые** — обновляются по ходу обсуждений, не раз в квартал. Если в чате договорились — пиши сюда в той же сессии.

## Как этим пользоваться

- **Перед написанием спека новой фичи**: проверь PILLARS, GLOSSARY, и есть ли релевантные DECISIONS / OPEN-QUESTIONS. Не плоди дубли решений.
- **После принятия дизайн-решения в чате**: записывай в DECISIONS.md одной строкой.
- **При появлении нового термина**: фиксируй в GLOSSARY.md, не давай ему расплыться.
- **При сомнении в направлении фичи**: проверь VISION и PILLARS. Если фича их не обслуживает — обсуди отдельно, прежде чем спекать.

## Что сюда **не** попадает

- Технические решения по архитектуре кода → `CLAUDE.md` или `docs/`.
- Конкретные численные балансы → `docs/balance.md` (когда появится) или `data/`.
- Имплементационные планы фич → `specs/NNN-name/`.
- Тех-долг → `docs/tech-debt.md` (когда появится).
