# Agents

Эта папка — **прототип системы агентов**. Каждый агент — `.md`-файл с инструкцией, которая меняет фрейм Claude (тон, фокус, запреты, обязанности) под конкретный тип задачи.

Не путать с `docs/design/` — там доки, которые Claude **читает как контекст**. Здесь файлы, которые **меняют поведение** Claude.

## Зачем это

Один большой `CLAUDE.md` под все типы задач = Франкенштейн с противоречивыми установками. Code review хочет придирчивость, design-doc keeping хочет роль скриба, balance tuning хочет числа а не архитектуру. Агенты разделяют эти режимы, чтобы каждый был чистым.

## Как это работает (роутинг)

**Явный роутинг** — основной механизм. `@agent-name` где угодно в сообщении:

```
@design-keeper зафиксируй, что мы решили скрывать прогресс аспектов
```

Claude читает `docs/agents/design-keeper.md`, переключается в этот фрейм, делает задачу. По окончании ответа возвращается в дефолтный режим (мозг). Не "залипает" в роли через turns.

Бонус: тот же `@`-синтаксис работает в Telegram-чатах команды и в GitHub PR/issues. Один меншн — три канала. Не нужно держать в голове три разных способа адресации.

**Без тэга — Claude спрашивает**, если запрос плаусибельно ложится на агента:

> Сделать как @design-keeper или @code-reviewer? Или ответить как мозг?

**Без тэга и без явного матча** — Claude отвечает как мозг (дефолтный режим, project instructions + CLAUDE.md). Не каждый запрос требует агента.

## Конвенции

### Имена агентов

- Короткие существительные с дефисами: `design-keeper`, `code-reviewer`, `balance-tuner`.
- Не CamelCase, не snake_case, не глаголы.
- Существительное описывает **роль**, не действие — потому что юзер обращается к агенту, а не командует им. `@reviewer` лучше чем `@review`.
- Гранулярность по ролям, иногда по функциям. Per-человек агенты не делаем (не `@nikita-mode`).

### Файловая структура

Каждый агент — один `.md`-файл по шаблону [`_template.md`](_template.md). Жёсткая структура:

- **Роль** — одно предложение.
- **Активируется когда** — триггеры (типы запросов, явный тэг).
- **Делает** — активные обязанности.
- **Не делает** — что отдаёт обратно мозгу.
- **Тон** — как говорит.
- **Хэнд-офф** — когда возвращает мозгу.

Шаблон стабильный, потому что на нём может потом быть линтер или авто-обёртка в Anthropic Skill.

### Брейн (default Claude)

"Мозг" — это дефолтный Claude этого проекта без агент-фрейма. Его инструкция — `PROJECT_INSTRUCTIONS.md` + корневой `CLAUDE.md`. Агенты — оверлеи поверх мозга. Где они конфликтуют — выигрывает агент (на время своей сессии).

Мозг видит больше контекста (стратегию, куда едет проект, скоуп). Агенты узко специализированы. Когда задача требует и стратегического суждения, и специализации — мозг рулит, иногда консультируется с инструкцией агента не переключаясь полностью.

## Что считать агентом, а что доком

Граница нечёткая, но рабочее правило:

- **Агент** = `.md`, который **меняет фрейм Claude** ("стань этим, не предлагай того"). Адресуется через `@`. Лежит здесь.
- **Док** = `.md`, который Claude **читает как контекст**, не меняя фрейма. Адресуется ссылкой. Лежит в `docs/design/`, `docs/balance/`, и т.д.

Тестовый вопрос: "если бы это было в `CLAUDE.md`, оно бы поменяло как Claude отвечает в принципе, или просто дало бы ему фактическую базу?". Первое — агент, второе — док.

## Как добавить нового агента

1. Скопируй [`_template.md`](_template.md) → `docs/agents/<имя>.md`.
2. Заполни секции. Особое внимание секции "Не делает" — она самая важная для предотвращения mission creep.
3. Добавь строку в таблицу "Текущие агенты" выше.
4. Скажи Claude явно: "теперь есть @<имя>" — он подтянет в next session-start.
5. Попробуй на реальной задаче. Если агент путается со старыми ролями или съезжает в сторону — поправь файл, итерируй.

## Что **не** делаем (пока)

- **Не оборачиваем в Anthropic Skill.** Пока проверяем паттерн на простых .md. Если выяснится, что Claude систематически забывает свериться с агентом — обернём, тогда триггеры будут авто.
- **Не делаем агента для каждой мелкой задачи.** Минимум — одна стабильная роль с регулярными обязанностями. Одноразовые задачи решаются мозгом.
- **Не плодим per-человека агентов.** Если для Никиты нужен особый стиль копирайтинга — это либо агент `narrative-writer` (по функции), либо просто промпт-стиль внутри обычного запроса. Не `@nikita`.

## Открытые вопросы по системе

- **Auto-loading при session-start** — стоит ли мозгу при старте сессии видеть список доступных агентов автоматически, или подгружать по запросу? Сейчас — вторая модель.
- **Агенты, вызывающие агентов** — может ли `@design-keeper` в процессе работы делегировать что-то `@code-reviewer`? Сейчас — нет, агент завершает задачу и возвращает мозгу, мозг роутит дальше.
- **Версионирование агентов** — если `design-keeper.md` поменяется, не поломает ли это уже идущие задачи? Пока ставка на то, что задачи короткие и редкие изменения агента не критичны.
