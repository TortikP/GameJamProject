# Localization Data

Generated by `pwsh -File scripts/tools/extract_localization.ps1`.

- `en.json` contains extracted English/source strings and generated placeholders for existing localization keys.
- `ru.json` contains the same keys with empty values, ready for translation.
- `_sources.json` maps every key back to the file/property/line it came from.

Runtime loading is handled by `scripts/infrastructure/localization.gd`. It registers these JSON files as Godot `Translation` resources through `TranslationServer`.

## Adding new localization keys

Any new localization key must be added to both localization files immediately:

- `en.json`
- `ru.json`

Do not leave a key only in code, scenes, configs, enemies, skills, maps, dialogues, or any other source file. If a new localized string is introduced, add its key to both JSON files in the same change.

Both files must always contain the same set of keys. This keeps the localization table, parser, and runtime translations in sync.

## Localization key naming rules

Use stable, descriptive, snake_case keys.

Prefer keys that describe where the text is used and what kind of text it is.

Good examples:

- `stapler_staple_shot_name`
- `stapler_staple_shot_desc`
- `stapler_staple_shot_tooltip`
- `status_burning_name`
- `status_burning_desc`
- `ui_main_menu_start_button_text`
- `ui_pause_restart_title`
- `dialogues_merchant_greet_text`

Avoid vague or unstable keys:

- `text1`
- `new_label`
- `button`
- `enemy_name`
- `tmp_desc`
- `test_string`

Keys should stay valid even if the displayed text changes later.

## Recommended key structure

Use prefixes by content type.

### Skills

```text
<skill_id>_name
<skill_id>_desc
<skill_id>_tooltip
```

Example:

```text
bee_sting_name
bee_sting_desc
bee_sting_tooltip
```

### Enemies

```text
<enemy_id>_name
<enemy_id>_desc
```

Example:

```text
burning_bear_name
burning_bear_desc
```

### Status effects

```text
status_<status_id>_name
status_<status_id>_desc
```

Example:

```text
status_poisoned_name
status_poisoned_desc
```

### UI strings

Use the screen, panel, or system name first, then the element purpose.

```text
ui_<screen_or_panel>_<element>_<property>
```

Examples:

```text
ui_main_menu_start_button_text
ui_settings_panel_language_label_text
ui_pause_restart_confirm
ui_map_editor_save_error
```

For common reusable UI labels, use:

```text
ui_common_<action_or_label>
```

Examples:

```text
ui_common_save
ui_common_cancel
ui_common_delete
```

### Dialogues

Use the dialogue group and line purpose.

```text
dialogues_<group>_<line_id>_text
dialogues_<group>_<line_id>_choices_<index>_label
dialogues_speakers_<speaker_id>_display_name
```

Examples:

```text
dialogues_merchant_greet_text
dialogues_merchant_greet_choices_0_label
dialogues_speakers_merchant_display_name
```

### Maps and games

```text
maps_<map_id>_name
games_<game_id>_name
games_<game_id>_levels_<index>_display_name
```

Examples:

```text
maps_sample_waves_name
games_sample_game_name
games_sample_game_levels_0_display_name
```

## Text value rules

When adding or editing localized values:

- Preserve placeholders exactly: `%s`, `%d`, `%.2f`, etc.
- Preserve placeholder order unless the formatting code explicitly supports reordering.
- Preserve BBCode tags such as `[b]`, `[color]`, `[/color]`.
- Preserve service symbols such as `★`, `→`, `—`, `…`, `II`, `✕`.
- Preserve intentional line breaks.
- Preserve HTML-like markup such as `<br>` when it is used by UI text.
- Do not translate internal ids, file names, enum values, or debug-only tokens unless they are meant to be shown to the player.

Examples:

```json
"ui_top_hud_turn": "Turn: %d",
"ui_skill_headline_damage": "%s — %d dmg",
"ui_game_editor_overwrite_body": "%s already exists.<br><br>Replace it?"
```

The Russian version must keep the same placeholders and markup:

```json
"ui_top_hud_turn": "Ход: %d",
"ui_skill_headline_damage": "%s — %d урн.",
"ui_game_editor_overwrite_body": "%s уже существует.<br><br>Заменить?"
```

### Skill tooltip variables

Skill tooltip strings (`<skill_id>_tooltip`) may use `$name$` variables. They
are resolved by `SkillFormatter.format_skill_human(skill)` from the current
skill config, after level scaling and parameter modifiers.

Common variables:

- `$damage$`, `$heal$`, `$cooldown$`, `$range$`
- `$radius$`, `$area_radius$`, `$max_chain_length$`, `$chain_targets$`
- `$duration$`, `$status_duration$`, `$status_value$`
- Status-specific variables from `data/status_effects/*.json:param_names`,
  for example `$n_attack_buff$`, `$n_attack_debuff$`,
  `$poisoned_damage_pct$`, `$burning_damage$`

Unknown variables are left unchanged so missing support is visible in-game.

## English and Russian file requirements

When adding a new key:

1. Add the key to `en.json`.
2. Add the same key to `ru.json`.
3. Put the English/source text in `en.json`.
4. Put the Russian translation in `ru.json`, or an empty string if the translation is not ready yet.
5. Re-run the extractor if the key was introduced through source content.

Valid temporary state:

```json
// en.json
"ui_new_feature_title": "New Feature"

// ru.json
"ui_new_feature_title": ""
```

Invalid state:

```json
// en.json
"ui_new_feature_title": "New Feature"

// ru.json
// missing key
```

## Workflow

After adding or renaming content defs, re-run the extractor:

```pwsh
pwsh -File scripts/tools/extract_localization.ps1
```

Then check that:

- `en.json` and `ru.json` contain the same keys.
- `_sources.json` points each extracted key to the correct source file/property/line.
- New keys follow the naming rules above.
- Placeholder sets match between languages.
- No player-facing text remains hardcoded outside localization.
