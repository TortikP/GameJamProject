# 007-skill-system — spec

**Owner:** Egor (claim-on-PR; «spell-craft / modifier engine» в CLAUDE.md рекомендован Sergey — coordinate перед merge)
**Status:** Superseded by [`021-skill-system-v2`](../021-skill-system-v2/spec.md) — schema расширения (loc keys, mood, level scaling, sound/animation), `entity` target → `actor`, `tags` → `behaviour_tags`. Контракт lifecycle (Skill → Ability[] → target × area × effect[]) сохраняется. Содержимое 007 — исторический baseline.

## Цель

Расширить существующий контракт абилки (THEME_PLAN.md §4, `scripts/core/abilities/`) до полноценной системы навыков:

- **Навык (Skill)** — новая надстройка: упорядоченная композиция из 1..N способностей, с общим cooldown.
- **Способность (Ability)** — существующее понятие, расширяется: добавляется **Area** между Target и Effect, и Effect становится **массивом** (1..N) вместо одного.
- **Эффекты** получают типизированную схему с обязательными полями `id`, `type`, `duration` и type-специфичными параметрами.
- **Модификаторы** теперь могут декорировать как способность, так и конкретный эффект, изменяя их **параметры** (а не только подключаясь к lifecycle-хукам).

Соответствует пилларсам:
- §1.5.2 Симметрия игрок ↔ монстр — навыки доступны и игроку, и AI; то же API.
- §1.5.1 Полная информация — превью урона/эффектов на ховере должно остаться рабочим (см. Out of scope: визуал — отдельно, но движок не должен ломать `predicted_damage_to`).

## Что меняется vs существующей системы

| Слой | Сейчас | После 007 |
|---|---|---|
| Skill | — | новый: `{id, cooldown, abilities: [Ability...]}` |
| Ability | `target × effect × modifier[]` | `target × area × effect[] × modifier[]` |
| Effect | один на способность | массив, каждый со своей схемой по `type` |
| Modifier | lifecycle-хуки (before/after_apply, after_cast) | параметр-мутаторы. Хуки убраны из системы, существующий `knockback` мигрирует в Move-эффект. |
| Cooldown | нет | поле на Skill |
| JSON | `data/abilities/*.json` (плоский) | `data/skills/*.json` (новый), `data/abilities/*.json` (расширяется) |

Существующие `data/abilities/*.json` мигрируются в новую схему как часть фичи (см. Out of scope: миграция инструментально — отдельный таск в `tasks.md`).

## Acceptance criteria

### Структура

- **AC-S1**: `Skill` имеет поля `id: StringName`, `cooldown: int`, `abilities: Array[Ability]` (длина ≥ 1).
- **AC-S2**: `Ability` имеет поля `id: StringName`, `target: AbilityTarget`, `area: AbilityArea`, `effects: Array[AbilityEffect]` (длина ≥ 1), `modifiers: Array[AbilityModifier]`.
- **AC-S3**: `id` способности и эффекта уникальны глобально (не в рамках навыка).
- **AC-S4**: Эффект имеет общие поля `id: StringName`, `type: StringName`, `duration: int`, `requires_alive_target: bool`, плюс type-специфичные (см. AC-E*).

### Типы целей (`AbilityTarget`)

Способность объявляет, какие категории целей допустимы.

- **AC-T1**: `entity` — игровой объект, способный к действиям (player, enemy, manekin).
- **AC-T2**: `hex` — клетка карты (координата `Vector2i`).
- **AC-T3**: `direction` — вектор от исполнителя.
- **AC-T4**: `object` — игровой объект без действий (ящик, колонна — будущая сущность).

### Области (`AbilityArea`)

- **AC-A1**: `self` — резолвится в [исполнитель], независимо от target.
- **AC-A2**: `chain` — поле `max_chain_length: int`. Стартовая цель = target. Каждая следующая цель — ближайший допустимый сосед предыдущей по гексам, ещё не затронутый этой способностью. Метрика — расстояние в гексах. Возврат к ранее затронутой цели запрещён. Цепочка обрывается, если на шаге N не нашлось соседа в радиусе 1 гекса.
- **AC-A3**: `zone` — гексы определённой формы (cone / circle / arc / line) от target. Цели в зоне сортируются от ближайшей к target к самой дальней.
- **AC-A4**: Пустой список целей → способность пропускается без ошибки, навык продолжается со следующей способности.

### Типы эффектов

Общие поля у всех: `id`, `type`, `duration`, `requires_alive_target`. Type-специфичные:

- **AC-E1**: `damage` — `damage: int`. `duration = 0` → мгновенный урон. `duration > 0` → DoT, тикает заданное число раз.
- **AC-E2**: `heal` — `heal: int`. `duration = 0` → мгновенный. `duration > 0` → HoT.
- **AC-E3**: `status` — `status: StringName` (id статуса). `duration` — длительность статуса в ходах. `duration = 0` бессмысленно (но допустимо — мгновенно навешен и снят, может быть полезно для триггеров).
- **AC-E4**: `move` — `move_type: StringName` (`push`/`pull`/`teleport`/...), `move_distance: int`. `duration = 0` — мгновенное перемещение. `duration > 0` — растянутое во времени (например, `pull` на 3 хода по 1 гексу).
- **AC-E5**: `create` — `game_object_id: StringName`. **Применим только если `Ability.target.kind = hex`**. Если целевой гекс занят сущностью или другим объектом → эффект пропускается без ошибки. Если способность с `create` повешена на target типа `entity`/`object`/`direction` → эффект пропускается без ошибки (но это валидно по схеме — не падать).

### Исполнение навыка

- **AC-X1**: Навык исполняется по способностям в порядке массива.
- **AC-X2**: Цели каждой способности резолвятся **на момент начала её исполнения** (не на момент старта навыка) — позиции, HP, состояние мира берутся свежими.
- **AC-X3**: Внутри способности — внешний цикл по целям, внутренний по эффектам: цель i получает все эффекты по порядку, потом цель i+1 получает все эффекты, и т.д.
- **AC-X4**: Цели в zone- и chain-области упорядочены от ближайшей к самой дальней.
- **AC-X5**: Перед применением каждого эффекта — проверка: если `requires_alive_target=true` и цель мертва, **этот** эффект пропускается для **этой** цели. Способность продолжает работу со следующей целью.
- **AC-X6**: Если цель умирает посреди обработки эффектов на ней, оставшиеся эффекты применяются с проверкой по AC-X5 (проверка на каждом эффекте).
- **AC-X7**: Смерть цели от эффекта одной способности **не прерывает навык**. Следующая способность исполняется как обычно (классический «вампиризм»: способность 1 = damage добивает; способность 2 = heal на исполнителя — обе применяются).
- **AC-X8**: Завершение последней способности → навык переходит на cooldown. Конкретная механика тика cooldown (start / end / per-turn) — out of scope этой спеки.

### Модификаторы

- **AC-M1**: Модификатор может декорировать способность.
- **AC-M2**: Модификатор может декорировать конкретный эффект внутри способности.
- **AC-M3**: Модификатор применим только если у декорируемого объекта существует параметр, который модификатор меняет. Иначе — отвергается без падения и без эффекта.
- **AC-M4**: Модификатор — **только параметр-мутатор**. Меняет числовое или категориальное поле способности/эффекта. Поведенческие хуки (lifecycle-listeners) в этой системе не предусмотрены — существующий `knockback_modifier.gd` мигрируется в самостоятельный effect типа `move(push)`.
- **AC-M5**: При нескольких модификаторах на одном параметре формула: `final = (base + Σadds) × Π muls`. Сначала суммируются все аддитивные бонусы (`+5 damage`, `+1 chain`), потом умножаются мультипликативные (`×1.5 damage`). Порядок добавления модификаторов в массив не влияет на результат — результат коммутативен по построению.
- **AC-M6**: Категориальные поля (`move_type`, `status`, `game_object_id`) модификатором заменяются (последний модификатор на этом поле выигрывает) — но в скоуп этой спеки такие модификаторы не входят, формализуется при необходимости.

### Acceptance scenarios (тесты)

1. **Чистый случай.** Skill = [Ability(self, [Damage(10, duration=0)])]. Сущность активирует → теряет 10 HP, навык на cooldown.
2. **Зональный конус с 3 целями.** Skill = [Ability(target=hex, area=cone(3), effects=[Damage(15), Status(stun, dur=2)])]. В конусе 3 врага A, B, C от ближнего к дальнему. Применить → A получает Damage, потом Status. Потом то же B, потом C. Порядок — внешний по целям.
3. **Вампиризм.** Skill = [Ability(target=entity, area=self_target, effects=[Damage(100)]), Ability(target=self, area=self, effects=[Heal(50)])]. Цель 50 HP. Активация → способность 1 убивает цель; способность 2 лечит исполнителя на 50.
4. **Move с requires_alive.** Effect Move(push, 2, requires_alive_target=true) идёт после Damage(20). Цель 10 HP. Damage убивает → Move на этой цели пропускается. Если в способности есть ещё живые цели — Move к ним применяется.
5. **Цепочка длины 3 при 4 кандидатах.** Chain(max=3) применён рядом с A; B сосед A, C сосед B, D сосед C. Цепь = [A, B, C], D не задет, повторов нет.
6. **Совместимый модификатор.** `+5 damage` на Damage(10) → урон 15.
7. **Несовместимый модификатор.** `+5 damage` на Heal(20) → отвергается, Heal работает как 20.
8. **Стек модификаторов.** На Damage(10) висят `+5 damage` и `×1.5 damage`. Итог: `(10 + 5) × 1.5 = 22.5` → округление по правилам типа (для int — floor, см. plan). Перестановка модификаторов в массиве не меняет результата.
9. **Create в занятый гекс.** Skill = [Ability(target=hex, effects=[Create(swarm)])]. Гекс занят врагом → эффект пропускается, ошибки нет.
10. **Create на entity-target.** Skill = [Ability(target=entity, effects=[Create(swarm)])] — конструктивно валидно, но при исполнении Create пропускается (нет гекса для спавна).

## Out of scope

- Активация навыка: валидация дальности, line of sight, очерёдность ходов, выбор цели игроком (UI). См. отдельную фичу активации.
- Стоимость навыка кроме cooldown (мана, очки действия) — пока только cooldown.
- Конкретные значения тика cooldown (per-turn / per-cast / start vs end resolution) — отдельный таск в plan.
- Прерывание навыка извне (стан исполнителя посередине) — отдельная фича.
- Анимации, VFX, звук, телеграфы.
- Конкретные статусы (что значит `stun`, `burn`...) — содержимое контента, не движок.
- Конкретные шаблоны для `create` (что такое `swarm`, `wall`...) — контент.
- UI выбора цели и превью эффектов.
- Балансные числа (damage, duration).
- Сериализация состояния навыков в save (cooldown timers).
- Миграция существующих `data/abilities/*.json` под новую схему — отдельный таск (`tasks.md`), не блокирует движок.
- **Триггерные / реактивные модификаторы**: `freeze_on_hit`, `ignite_on_hit`, `extra_cast` и подобные «при срабатывании одного эффекта добавить второй» / «повторить с задержкой». Не выражаются через параметр-мутацию, требуют отдельной системы триггеров — отдельная фича (потенциально spec-008), если станет нужно. Кейс `heal_caster` (вампиризм) сюда **не входит** — он уже выражается через 2 способности в одном навыке (см. AC-X7, scenario 3).
- Координация с Sergey по «spell-craft / modifier engine» — на стороне команды, не в спеке.

## Закрытые вопросы (clarify-лог)

- **Q-MOD-1 → (a)** «`(base + Σadds) × Π muls`» — стандарт RPG-аффиксов. Зафиксирован в AC-M5.
- **Q-MOD-2 → (a)** «только мутаторы» — поведенческих хуков нет в этой системе. Зафиксирован в AC-M4. Триггерные модификаторы (`freeze_on_hit`, `extra_cast`) явно вынесены в Out of scope — если станут нужны, делаем отдельной фичей. Вампиризм (`heal_caster`) — не модификатор, это вторая способность в навыке.

## Заметки реализации (для будущей plan.md)

- Существующий `Ability.cast()` в `scripts/core/abilities/ability.gd` уже почти подходит как «исполнитель одной способности» — нужно: добавить `area`, преобразовать `effect` в `effects[]`, перенести цикл целей наружу, цикл эффектов — внутрь.
- Новый `Skill` — тонкая обёртка: `cast()` дёргает `cooldown`-проверку, потом по очереди вызывает `Ability.cast()` каждой способности.
- `EventBus.ability_cast` остаётся; добавится `EventBus.skill_cast` (для UI cooldown / preview).
- `predicted_damage_to` (для preview, см. CLAUDE.md §1.5.1) пересматривается: суммирует по всем Damage-эффектам всех способностей с учётом активных модификаторов.
- `THEME_PLAN.md §4` пересматривается под новый контракт в том же PR (документ — не код, миграция тривиальная).
